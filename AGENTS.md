# Инструкции
После выполнения задания - обязательно запускай сайт в эмуляторе и тестируй все страницы сайта.
Если перед запуском требуется обновить БД - сообщи мне об этом и дай команду для artisan.

# Программные требования.
Сайт на английском языке (EN-GB), весь контент тоже на английском языке.
Не используем node.js.
Каждый блок кода должен быть прокомментирован на английском языке.
Все обязательные поля отмечены в ТЗ как обязательные, остальные не обязательные. Но технически обязательные (типа ID, etc.) все равно должны быть обязательны. На всех обязательных полях должна быть валидация.
При отсутствии данных в БД ничего не должно крашится, должно выводится сообщение об отсутствии данных.
Сайт не должен быть доступен для индексирования поисковиками вообще.

# Визуальные требования.
При кодинге view обязательно использование классов Bootstrap 5.x.
У каждой страницы сайта должен быть единый шаблон footer.
Каждое действие в системе должно логироваться с указанием пользователя, где произведено действие и какое произведено действие.
Переход к мобильной версии начинается так: lg и больше - десктоп, md и меньше - мобильные, т.е. там, где есть колонки, будет col-lg-x.
Все таблицы должны иметь пагинацию по 20 результатов в стиле bootstrap.

# Авторизация
Для доступа к любому разделу сайта необходимо пройти авторизацию (любого типа).
1. Персональная авторизация (по логину паролю).
Обычная стандартная авторизация. Авторизация происходит по почте и паролю.
2. Публичная авторизация.
Публичная авторизация (гостем) доступна только для кейса со status=progress.
Каждый кейс должен иметь свой уникальный сложный public_link, длиной 8–16 символов, который позволяет просматривать страницу конкретного кейса и больше ничего. При заходе по этой ссылке система должна спрашивать Postal Code и если он введен правильно, то показываеть этот кейс. При неправильном вводе должна выдавать ошибку. Нужна простейшая защита от брутфорса (любая, хоть блокировка IP на какое-то время при неправильном вводе, хоть что-то другое не сильно сложное).
При публичной авторизации Гость не имеет право ничего менять в системе, но он может просматривать страницу кейса (Стадии, Задачи, Чат) и скачивать в чате файлы.

# Пользователи
Пользователи имеют 3 типа (role):
1. Пользователь admin.
Admin имеет неограниченные права в системе.
2. Пользователь legal.
Он же юрист, он же solicitor. Привязывается к кейсу админом, имеет доступ только к тем кейсам, в которых он является юристом.
3. Пользователь client.
Клиент компании.
